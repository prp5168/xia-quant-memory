## 记录系统修复（2026-03-25）
- 背景：检查 observe-log / forecast-log / watchlist 后，确认当前链路整体在跑，但发现两个明确的数据质量问题：
  1. `observe-log.jsonl` 里缺少 `dayDiff` 字段；
  2. `watchlist.json` 的 prune 与新增使用了不同时间口径，导致已到 Wellington 本地次日时，部分美洲城市的 2026-03-24 项会出现“先删后又按各城市 local today 逻辑加回”的现象。
- 已对 `scripts/sim_portfolio_scan.mjs` 做两处功能修复：
  1. 在 `observeSnapshots.push(...)` 中补写 `dayDiff: getLocalDayDiff(date, st)`，使 observe-log 与 forecast-log 后续都可按 dayDiff 做一致切片分析；
  2. 将 `pruneWatchlist` 改为按每条 watchlist 对应城市自己的本地日期判断是否过期，不再使用 Wellington/最先进时区作为全局 prune 基准。
- 当前判断：
  - `marketMissing` 占位逻辑代码本身存在，不是漏写；
  - 历史早期 observe / forecast 不完全对齐的问题仍保留为历史遗留数据现象，本轮未回填旧日志，只修复未来新增记录的一致性。
- 影响：
  - 后续 observe-log 可直接用于按 today/tomorrow/dayDiff 做切片；
  - watchlist 的保留/清理逻辑将与天气市场“按城市本地时间推进”的结算口径一致，减少脏名单。
- 补充：在修复后做 `node --check` 时，额外暴露出脚本文件尾部残留了一段重复脏文本（重复的 `main().catch(...)` 和孤立的 `.compactActionLines].join(...)` 片段），属于现存语法错误；已一并删除，恢复脚本可执行状态。
- 待验证：下一轮扫描后，确认 observe 最新记录已带 `dayDiff`，且 watchlist 不再因跨时区口径混用而留下不合理过期项。
